## Scope Note

This crosswalk compares BFO's `Process`, DOLCE's `Process` (a subtype of `Stative`), UFO's informal "process" notion, and SUMO's `Process` (sibling of `Object` under `Physical`), completing the pair started with `docs/crosswalks/concepts/event.md` and directly resolving that crosswalk's `uncertainty-004`.

**The asymmetry runs in the opposite direction from `event.md`, and just as cleanly.** There, BFO and SUMO lacked a top-level "Event" category while DOLCE and UFO had one. Here, it's the reverse: BFO and SUMO both treat `Process` as a genuine top-level formal category, while DOLCE nests `Process` one level down as a subtype of `Stative` (alongside `State`), and **UFO has no distinct formal `Process` class at all** — "process" appears in UFO/UFO-B literature only informally, for "the life of an endurant" (the sum of events that manifest an endurant's dispositions), with `Event` serving as UFO's actual primary perdurant category. Read together, `event.md` and this crosswalk show that each of the four sources picks a different primary term for the perdurant-with-duration territory: BFO picks `Process`, SUMO picks `Process`, DOLCE keeps both `Process` and `Event` as siblings one level down, and UFO picks `Event` alone.

**Confirmation (2026-07-01), following direct primary-source verification:** unlike `object.md`'s correction (where "UFO has no `Object` class" turned out to be wrong), this crosswalk's claim about `Process` is confirmed correct after fetching and manually extracting text from UFO's actual 2021 Applied Ontology paper. The primary source is explicit that "process" is a deliberately informal label, not a distinct formal class: "there is a special perdurant (which we may call a process, in a very particular sense) that is the life of an endurant." The paper's own worked example reinforces this: its "Jogging Process" case (the maximal sum of "Jogging Event"s manifesting a jogger's disposition) is formalized in the axioms as `EventType(JoggingProcess)` — i.e., `JoggingProcess` is typed *under* `EventType`, not under any separate `ProcessType`. No axiom or definition anywhere in the paper introduces a class literally named `Process`. See Source Definitions and Uncertainty for the verified picture.

Exclusions: this crosswalk does not re-litigate Event (see `event.md`) and does not attempt UFO-B's disposition/manifestation theory in full.

**Update (2026-07-02), ADR-0015 buildout:** extended to GFO, YAMATO, TUpper, and GUM. Unlike UFO, all four new sources positively name a Process-equivalent term, so the crosswalk's central "some sources lack a distinct Process category" asymmetry is a BFO/DOLCE/SUMO/UFO-specific finding, not one that recurs among the newly added sources. GFO's `Process` is a genuine top-level formal category (subclass of `Occurrent`, explicitly `owl:disjointWith` `Change` and `History`) with a positive criterion ("directly in time," not reducible to a collection of time boundaries) — the closest structural match to BFO's Process of any new source. YAMATO's 2010 technical report does not subdivide its `Occurrent` class further in the passages reviewed; "process" there is essentially synonymous with the top-level Occurrent(Process) label already used in `continuant-occurrent.md`, not a distinct finer-grained class the way GFO/BFO have one. TUpper's `activity` (the repeatable type) is a different correspondence than `event.md`'s `activity_occurrence` (a token occurrence of that type) — a type/token pair without a clean BFO:Process analog, since BFO's Process is itself a particular, not a type. GUM's own class is literally named `Process`, the closest terminological match of any new source, but its criterion (verb-realizability in systemic-functional grammar) is linguistic rather than temporal/metaphysical.

**Epistemic note:** BFO, DOLCE, UFO, and SUMO content is all now verified directly against primary sources (`bfo-core.ttl`, WonderWeb D18, the 2021 Applied Ontology paper, and `Merge.kif`, respectively). All four extractions reuse the same manual `zlib`-stream-decompression technique, since no PDF-text-extraction tool is otherwise available in this environment.
